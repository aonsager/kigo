# Kigo — agent guide

Domain vocabulary lives in [CONTEXT.md](./CONTEXT.md); the goal state and its
evidence procedures live in [docs/GOAL.md](./docs/GOAL.md). This file is the
operational guide for any agent (especially the afk-loop implementer) touching
the code: how to build, how to test, and the traps that waste whole iterations.

## Stack

SwiftUI app + WidgetKit extension + StoreKit 2. Swift 6 / **Xcode 26.5** (see the
toolchain note under Build & test — the simulator *runtime* pin is still 26.4.1).
The Xcode project is generated by **XcodeGen** from `project.yml` and is
**gitignored** — never edit `Kigo.xcodeproj` by hand; edit `project.yml` and
regenerate.

## Build & test

Always regenerate the project first, then run the hardened test command. Do
**not** run a bare `xcodebuild test` — the extra flags below are load-bearing
(see "Why the flags" — they are what turns a hung test into a fast failure).

```bash
xcodegen generate
# 12-min inner cap (perl alarm — macOS has no coreutils `timeout`) so a hung
# command fails fast instead of wedging the whole afk iteration; XCTest
# execution-time allowances force a single stuck test to fail at 120s.
perl -e 'alarm shift; exec @ARGV' 720 \
  xcodebuild test \
    -scheme Kigo \
    -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.4.1' \
    -test-timeouts-enabled YES \
    -default-test-execution-time-allowance 120 \
    -maximum-test-execution-time-allowance 300 \
    CODE_SIGNING_ALLOWED=NO \
    -only-testing:KigoTests/<SuiteName>
```

- Pass `** TEST SUCCEEDED **` (exact string) as the success signal. Drop
  `-only-testing:` to run the whole suite.
- **Pin the runtime: `OS=26.4.1`** (the installed iOS 26.4 point release — check
  with `xcodebuild -showdestinations`). Bare `name=iPhone 17` resolves
  nondeterministically across installed runtimes (26.2 / 26.4.x). A stale pin
  fails *fast and legibly* in ~60s (`Unable to find a device matching…`), never
  hangs. The point of pinning is to stay off the **iOS 26.5** runtime — see the
  StoreKit trap below.
- **Toolchain note (Xcode 26.5, since 2026-06-18).** Xcode was upgraded 26.4.1 →
  **26.5** to fix the Xcode 26.4 `simdiskimaged` crash that repeatedly wedged the
  loop under UI-test load (CoreSimulatorService dies under sustained `xcodebuild
  test`; verified fixed on 26.5, even on macOS 26.3.1 — no macOS upgrade needed).
  Two non-obvious consequences:
  - **Do *not* bump the runtime pin to 26.5.** The pin stays `OS=26.4.1` so tests
    keep running on the iOS 26.4 runtime, off the iOS-26.5 StoreKit-broken runtime
    (ADR 0009 / the StoreKit trap below). Upgrading the *toolchain* and bumping the
    *runtime pin* are independent — only the toolchain moved.
  - **The iOS 26.5 simulator runtime must stay installed** (Xcode → Settings →
    Components). Xcode 26.5 refuses to enumerate *any* iOS simulator destination
    until its own platform is present; with it installed, 26.2 / 26.4.1 / 26.5 all
    resolve and the 26.4.1 pin works. It is installed but never the test target.
  - Rollback if ever needed: `Xcode-26.4.1.app` is kept alongside —
    `sudo xcode-select -s /Applications/Xcode-26.4.1.app/Contents/Developer`.

### Why the flags (this is not optional ceremony)

A test that *hangs* (an `await` that never resolves, a network/StoreKit call,
a stuck simulator) otherwise consumes the entire 45-min afk iteration, gets
SIGALRM-killed (`rc=142`), and repeats — burning the loop's whole budget with no
diagnostic. The flags convert that into a legible failure:

- `-test-timeouts-enabled YES` + `-default-test-execution-time-allowance 120`
  fails any individual test that runs longer than 120s, with a stack trace.
- the `perl alarm` wrapper bounds everything *outside* a single test (xcodegen,
  simulator boot, linking) so even those fail fast.

If you find yourself waiting minutes for a test command with no output, it is
hanging — kill it and treat it as a failure; do not wait it out.

## Test-output gotchas

- **Phantom "0 tests".** An empty XCTest runner can print `Executed 0 tests`
  while the real result is elsewhere. Trust the `** TEST SUCCEEDED **` /
  `** TEST FAILED **` line and the per-suite count, not a bare "0 tests".
- The `slice-gate` GitHub check is a deliberate no-op (always green). Real
  verification is the in-session test run; do not read meaning into it.

## StoreKit testing — the trap that wedged this loop once

**Do not gate any criterion on driving a real purchase through `SKTestSession`
under `xcodebuild test` from the CLI.** On recent simulator runtimes (notably
iOS 26.5) the CLI does not push the `.storekit` configuration to the simulator's
`storekitd`, so `SKTestSession.buyProduct(...)` either throws
`SKInternalErrorDomain Code=3` or silently falls through to the production App
Store and **hangs indefinitely**. It only works through Xcode's IDE launch path
(Cmd+U), which the public CLI cannot invoke. This is an Apple platform defect,
not something an implementer can code around — it is what burned three 45-min
iterations on the C6 entitlement slice.

**The pattern instead — wrap StoreKit behind an injectable protocol:**

- `EntitlementProvider` depends on a small `EntitlementTransactionSource`
  protocol (`func activeProductIDs() async -> Set<String>`).
- Production conforms via StoreKit 2's `Transaction.currentEntitlements`
  (a thin adapter, correct by inspection).
- **Tests inject an in-memory fake** and feed it product IDs — verifying the
  entitlement *logic* (active/inactive derivation) headlessly, deterministically,
  in milliseconds, with no `storekitd`, no simulator purchase, no network.

The real StoreKit adapter, if you want it exercised end-to-end, gets at most a
**single** `SKTestSession` integration test marked non-blocking (run in the Xcode
IDE / a manual lane), never on the loop's gating path. See ADR 0009 and the C6
criterion in `docs/GOAL.md`.

## Watching the loop

`afk-run.sh`'s `wrapper.log` only shows per-iteration start/done lines — the
slice work runs as an Agent-tool subagent inside each headless session, so its
blow-by-blow is persisted to `~/.claude/projects/<repo-slug>/<uuid>.jsonl` (the
orchestrator) and `<uuid>/subagents/agent-*.jsonl` (the worktree work), not to
`wrapper.log`. `scripts/afk-tail.py` (read-only) follows both and renders them
in the wrapper's `TS |   · …` format, nesting subagent lines as `↳`. Use it via:

```bash
./scripts/afk-watch.sh          # run the loop + follow it live (Ctrl-C stops both)
./scripts/afk-tail.sh           # just follow (loop already running elsewhere)
tail -f .afk/stream.log         # ...or tail the persisted render from anywhere
```

## Screenshot evidence — the `afk-evidence` release must pre-exist

The loop publishes UI screenshots by uploading them as assets on a GitHub
**release tagged `afk-evidence`** (a pre-release) and inline-embedding the
download URL (`…/releases/download/afk-evidence/slice-<N>-<desc>.png`) in the
issue/PR comment. The repo is public, so `![…]` embeds render.

**The trap:** *creating* a release is a sensitive outward-facing action the
harness permission classifier declines autonomously, but *uploading an asset to
an existing release* is allowed. So if `afk-evidence` does not yet exist, the
very first slice's screenshot fails to land — the comment falls back to a text
"⚠️ visual evidence did not land" explanation (this is exactly what happened to
#85; the PNG was recovered post-hoc from the `.xcresult` in DerivedData via
`xcrun xcresulttool export attachments` and uploaded once the release existed).

**Therefore: pre-create the `afk-evidence` release during `afk-init`** (e.g.
`gh release create afk-evidence --prerelease --notes "afk loop screenshot
evidence"`) so the per-slice upload path is never blocked mid-loop. Naming
convention for assets: `slice-<N>-<short-desc>.png`.
